ü™§ The Coding-First Trap

New feature request. You read the ticket, open your IDE, and start typing. Halfway through, you realize your approach won't work. Start over. Repeat until something works. Tech debt accumulates.

‚ùå Junior Approach

  1. Read ticket
  2. Open IDE
  3. Start coding
  4. Realize approach won't work
  5. Start over
  6. Spend days fixing edge cases

Result: 2 weeks, painful code reviews, tech debt created

‚úÖ Senior Approach

  1. Read ticket, ask clarifying questions
  2. Draw the data flow
  3. Identify edge cases on paper
  4. Write approach in plain English
  5. Get quick feedback
  6. Code with clear plan

Result: 1 week, smooth reviews, minimal tech debt

üìê The Design-First Workflow

Before you write a single line of code, complete these five steps:

1
~30 minutes

Clarify

What EXACTLY is the requirement? What's the scope? What's NOT included? What are the success criteria?

2
~1 hour

Diagram

Data flow: What data goes where? Component diagram: What pieces are involved? Sequence: What happens in what order?

3
~30 minutes

Edge Cases

What if input is invalid? What if external service fails? What if user does something unexpected? Race conditions?

4
~30 minutes

Approach Doc

Plain English explanation. Key decisions and why. Risks and mitigations.

5
~15 minutes

Quick Review

Show a senior your plan. Get feedback BEFORE you code. Adjust based on their experience.

‚è±Ô∏è Total: ~2.5 hours of planning

This "wastes" half a day. But it saves you 3-5 days of rework. You spend less total time AND produce better code.

ü§ñ Design + AI = Superpowers

Here's the secret: AI is amazing at coding but terrible at design. Give AI a clear design, and it executes beautifully. Give it vague requirements, and it gives you vague code.

‚ùå Bad AI Prompt

"Write code to handle payments"

‚Üí Generic code, misses edge cases, won't fit your architecture

‚úÖ Good AI Prompt

"Here's my design doc with sequence diagram. Here are the 5 edge cases. Write the PaymentProcessor class that handles this flow."

‚Üí Exactly what you need, handles edge cases, fits your architecture

Example AI Prompt 
# Context for AI:

I need to implement a payment processing feature.

## Design Doc Summary
- User initiates payment via API endpoint POST /payments
- Validate amount > 0, user has valid card on file
- Call Stripe API with retry (max 3 attempts, exponential backoff)
- On success: create Transaction record, update user balance
- On failure: log error, return appropriate error code

## Edge Cases to Handle
1. Stripe timeout ‚Üí retry with backoff
2. Stripe rate limit ‚Üí queue and retry later
3. Duplicate payment attempt ‚Üí idempotency key
4. Card declined ‚Üí return user-friendly error
5. Partial failure (charge succeeded but DB write failed) ‚Üí reconciliation

## Our Patterns
- Use @retry_with_backoff decorator from our utils
- Use our BaseAPIError for consistent error responses
- Follow existing PaymentService pattern in services/

# Now write the PaymentService.process_payment method.
